## Decision Summary

**Implementing Approach C: Type-test assertions + `.passthrough()` + expanded contract tests.** Fixes active data loss in API response parsing, adds compile-time drift detection between the upstream `lib/payclaw.ts` types and local `lib/payclaw/schemas.ts` Zod schemas, and expands the existing contract test pattern for the Convex→API boundary.

### Step 1.1: Remove `wex` from PaymentMethod Schema

The upstream `PaymentMethod` type (`lib/payclaw.ts:53–56`) only supports `{ type: 'brex', cardId: string }`. The `wex` variant in the local schema has no upstream support. If the LLM selects it, Zod validates it as acceptable, `client.ts` sends it to the API, and the API rejects it — a runtime failure path that currently exists in production.

In `lib/payclaw/schemas.ts`, replace the `paymentMethodSchema` (lines 17–22):

```typescript
// lib/payclaw/schemas.ts — replace lines 17–22
export const paymentMethodSchema = z
  .object({
    type: z.literal('brex'),
    cardId: z.string().min(1),
  })
  .describe('Payment rail override supported by PayClaw.')
```

With only one variant, a discriminated union adds no value. If upstream adds new payment types, the type-drift test (Phase 2) will flag it, and the union can be reintroduced then.

### Step 1.2: Add `.passthrough()` to Response-Parsing Schemas

Add `.passthrough()` to every Zod schema that parses data FROM the PayClaw API. This stops Zod from silently stripping fields that the upstream type declares but the local schema doesn't.

Currently `jobSchema` declares 11 fields; the upstream `Job` type has 17. Six fields — `vendorId`, `vendorSlug`, `targetUrl`, `shippingAddress`, `paymentMethod`, `browserProvider` — are silently dropped on every `getJob()` call (`client.ts` line 256). This is active data loss.

**Rule**: Response schemas (API → app) get `.passthrough()`. Input schemas (LLM → app) do NOT — stripping hallucinated fields from LLM output is correct behavior.

|--------|-----------|-------------------|
| `shippingAddressSchema` | Input (LLM → app) | No |
| `paymentMethodSchema` | Input (LLM → app) | No |
| `payClawToolInputSchema` | Input (LLM → app) | No |
| `reportedDataSchema` | Response (nested) | No (already uses `.catchall()`) |
| `jobResultSchema` | Response (API → app) | **Yes** |
| `jobSchema` | Response (API → app) | **Yes** |
| `jobEventSchema` | Response (API → app) | **Yes** |
| `createJobResponseSchema` | Response (API → app) | **Yes** |
| `eventsListResponseSchema` | Response (API → app) | **Yes** |
| `sseStreamDoneSchema` | Response (API → app) | **Yes** |

### Step 2.1: Create the Test File

Create `lib/payclaw/__tests__/type-drift.test.ts`.

This file uses Vitest's `expectTypeOf` for type-level assertions. These are checked during `bun run typecheck` (compile-time) and appear as passing tests during `bun run test` (runtime no-ops). When the sync script pulls upstream changes that alter type shapes, this file fails to compile, making drift immediately visible.

**Assertion strategy**:
- **Bilateral `toEqualTypeOf`** for types that cross trust boundaries in BOTH directions (app → API for sends, API → app for receives). Catches new optional fields that one-way `satisfies` misses.
- **One-way `toMatchTypeOf`** for response-only types where the local schema is an intentional superset (extra optional fields, nullable wrappers).
- **Per-field checks** for `Job`, which has intentional structural deviations (missing fields preserved via `.passthrough()`).

```typescript
// lib/payclaw/__tests__/type-drift.test.ts
//
// Compile-time assertions detecting structural drift between the auto-synced
// upstream types (lib/payclaw.ts) and local Zod schemas (lib/payclaw/schemas.ts).
//
// Runs during both `bun run typecheck` and `bun run test`.
// When this file fails to compile, `scripts/sync-payclaw-client.sh` pulled
// upstream changes that require updating the local Zod schemas.

import { describe, it, expectTypeOf } from "vitest"

import type {
  ShippingAddress as UpstreamShippingAddress,
  PaymentMethod as UpstreamPaymentMethod,
  JobStatus as UpstreamJobStatus,
  JobResult as UpstreamJobResult,
  Job as UpstreamJob,
  JobEvent as UpstreamJobEvent,
} from "@/lib/payclaw"

import type {
  ShippingAddress as LocalShippingAddress,
  PaymentMethod as LocalPaymentMethod,
  JobStatus as LocalJobStatus,
  JobResult as LocalJobResult,
  Job as LocalJob,
  JobEvent as LocalJobEvent,
} from "@/lib/payclaw/schemas"

// ---------------------------------------------------------------------------
// Bilateral checks — types that cross trust boundaries in BOTH directions.
// Uses toEqualTypeOf for strict structural equality. Catches:
//   - Field added to either side (including optional fields)
//   - Field removed from either side
//   - Field type changed on either side
// ---------------------------------------------------------------------------

describe("bilateral type drift (app ↔ API)", () => {
  it("ShippingAddress: local ↔ upstream are structurally equivalent", () => {
    expectTypeOf<LocalShippingAddress>().toEqualTypeOf<UpstreamShippingAddress>()
  })

  it("PaymentMethod: local ↔ upstream are structurally equivalent", () => {
    expectTypeOf<LocalPaymentMethod>().toEqualTypeOf<UpstreamPaymentMethod>()
  })

  it("JobStatus: local ↔ upstream are structurally equivalent", () => {
    expectTypeOf<LocalJobStatus>().toEqualTypeOf<UpstreamJobStatus>()
  })
})

// ---------------------------------------------------------------------------
// One-way checks — types that flow FROM the API only (responses).
// Local schemas are intentionally more permissive (extra optional fields,
// nullable wrappers, .passthrough() index signatures). We check that
// upstream's shape is parseable by our schema, not the reverse.
// ---------------------------------------------------------------------------

describe("one-way type drift (API → app responses)", () => {
  it("JobResult: upstream is assignable to local (local is a superset)", () => {
    // Local adds: credentialIds, orderNumber, reportedData, nullable wrapper.
    // Local has .passthrough() adding an index signature.
    // Upstream's shape must still be accepted by local's parser.
    expectTypeOf<UpstreamJobResult>().toMatchTypeOf<NonNullable<LocalJobResult>>()
  })

  it("JobEvent: upstream is assignable to local", () => {
    expectTypeOf<UpstreamJobEvent>().toMatchTypeOf<LocalJobEvent>()
  })

  // Job has intentional deviations:
  // - Local omits 6 fields: vendorId, vendorSlug, targetUrl, shippingAddress,
  //   paymentMethod, browserProvider (preserved at runtime via .passthrough())
  // - Local makes steps optional (upstream declares required)
  // Full structural match is intentionally NOT checked. Per-field checks on
  // shared fields catch type changes and removals.
  it("Job: shared fields are type-compatible", () => {
    expectTypeOf<UpstreamJob["id"]>().toEqualTypeOf<LocalJob["id"]>()
    expectTypeOf<UpstreamJob["status"]>().toEqualTypeOf<LocalJob["status"]>()
    expectTypeOf<UpstreamJob["createdAt"]>().toEqualTypeOf<LocalJob["createdAt"]>()
    expectTypeOf<UpstreamJob["vendorUrl"]>().toEqualTypeOf<LocalJob["vendorUrl"]>()
    expectTypeOf<UpstreamJob["vendorName"]>().toEqualTypeOf<LocalJob["vendorName"]>()
    expectTypeOf<UpstreamJob["maxSpend"]>().toEqualTypeOf<LocalJob["maxSpend"]>()
  })
})

// ---------------------------------------------------------------------------
// Structural inventory — documents intentional deviations.
// Review this section when upstream types change.
// ---------------------------------------------------------------------------
// Upstream Job fields NOT in local jobSchema (preserved via .passthrough()):
//   - vendorId?: string
//   - vendorSlug?: string
//   - targetUrl?: string
//   - shippingAddress?: ShippingAddress
//   - paymentMethod?: PaymentMethod
//   - browserProvider?: string
//
// Upstream Job.steps is required (AgentStep[]); local is optional.
// Kept optional for defensive parsing — API sometimes omits steps.
//
// Local JobResult extras NOT in upstream:
//   - credentialIds?: string[]
//   - orderNumber?: string
//   - reportedData?: ReportedData
//   - .nullable() wrapper
//
// Local PayClawApiError (lib/payclaw/client.ts:27–38) is an independent
// reimplementation of upstream PayclawApiError (lib/payclaw.ts:148–158).
// Different name casing, different fields. This is intentional — the local
// client owns its own error contract.
```

### Step 2.2: Handle `toEqualTypeOf` Fallback

Vitest's `toEqualTypeOf` is stricter than `toMatchTypeOf` — it checks exact structural equality. If Zod's inferred types have minor representation differences from hand-written TypeScript types (e.g., `string | undefined` vs implicitly optional), `toEqualTypeOf` may produce false positives.

If any bilateral assertion fails on a false positive after Phase 1 is complete:

1. Inspect the error message to identify the structural mismatch.
2. If the mismatch is a Zod inference artifact (not a real drift), downgrade that specific assertion to bidirectional `toMatchTypeOf`:

```typescript
// Fallback: bidirectional toMatchTypeOf if toEqualTypeOf has false positives
expectTypeOf<LocalShippingAddress>().toMatchTypeOf<UpstreamShippingAddress>()
expectTypeOf<UpstreamShippingAddress>().toMatchTypeOf<LocalShippingAddress>()
```

Bidirectional `toMatchTypeOf` catches everything except upstream adding a new optional field — an acceptable gap since `.passthrough()` preserves those fields at runtime for response types, and for input types (ShippingAddress, PaymentMethod), the LLM would need to be instructed to populate new fields regardless.


If any assertion fails, it means Phase 1 corrections didn't fully align the schemas. Fix the schema in `lib/payclaw/schemas.ts`, not the test.

## Critical Implementation Patterns

### `.passthrough()` Placement Rule

**Response schemas** (parse external API data): Add `.passthrough()`. Stripping unknown fields from API responses causes silent data loss — the upstream type may declare fields that the local schema doesn't enumerate.

**Input schemas** (validate LLM tool output / form data): Do NOT add `.passthrough()`. Stripping unknown fields from untrusted input is correct defense-in-depth.

### Type-Test Assertion Strategy

| Type | Direction | Assertion | Why |
|------|-----------|-----------|-----|
| `ShippingAddress` | Bilateral | `toEqualTypeOf` | Sent to API AND received from API |
| `PaymentMethod` | Bilateral | `toEqualTypeOf` | Sent to API in tool input |
| `JobStatus` | Bilateral | `toEqualTypeOf` | Used in both queries and responses |
| `JobResult` | Response only | `toMatchTypeOf` (upstream → local) | Local is intentional superset |
| `JobEvent` | Response only | `toMatchTypeOf` (upstream → local) | Structurally equivalent today |
| `Job` | Response only | Per-field `toEqualTypeOf` | Local intentionally omits 6 fields |

**`toEqualTypeOf` vs `toMatchTypeOf`**: `toEqualTypeOf` catches new optional fields; `toMatchTypeOf` does not. Use `toEqualTypeOf` for bilateral types where any structural change (even adding an optional field) signals a review is needed. Fall back to bidirectional `toMatchTypeOf` only if Zod inference artifacts cause false positives.

### Error Propagation

| Failure Point | Behavior |
|---------------|----------|
| Type-drift test fails during `typecheck` | Build blocked; developer updates Zod schemas to match upstream |
| Contract test fails during `test` | Test suite blocked; developer fixes the Convex→API mapping or schema |
| Sync script typecheck warns | Warning printed; file still synced; developer investigates |
| `.passthrough()` causes consumer type error | Fix at consumer site by picking fields explicitly instead of spreading |

### Evaluated and Rejected Alternatives

**Option A: Re-export + `satisfies` Guards.** Creates a bridge module re-exporting upstream types, with `satisfies` constraints on Zod schemas. Rejected because: `satisfies` only checks one-way assignability and misses upstream adding optional fields. Would also introduce new coupling to `lib/payclaw.ts` from production code (currently nothing imports it).

**Option B: Canonical Zod Bridge with Derived Everything.** Single bridge Zod schema constrained against upstream, with all downstream consumers deriving via `.extend()/.pick()/.omit()`. Rejected because: (1) Convex functions cannot import from `lib/`, making the "shared field metadata" proposal physically impossible. (2) `.describe()` annotations for LLM tool calling cannot be auto-derived. (3) Derivation chains create implicit coupling that's harder to debug than independent schemas.
