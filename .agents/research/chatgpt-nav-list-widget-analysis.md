## Executive Summary

ChatGPT's clickable card carousel (`data-testid="nav-list-widget"`) is **not** HTML generated by the model. It is a React component that renders structured metadata attached to the conversation object by the backend. The model produces text with citation markers; the backend attaches `search_result_groups` metadata containing URLs, titles, images, dates, and domain information; the frontend reads this metadata and renders the card carousel, inline citation pills, and sources button as separate UI treatments of the same underlying data.

**Confidence**: The field name `search_result_groups` is **Confirmed** from HAR observation [^1]. The internal structure of this field (specific sub-fields like image URLs, dates, etc.) is **Inferred** from the rendered HTML output — the HAR study confirms the field exists in message metadata but does not expose its internal schema.

## 1. What the User Sees

Three distinct citation surfaces appear in a single ChatGPT response that used web search:

| Surface | Location | Purpose |
|---|---|---|
| **Inline citation pills** | Embedded within paragraph text | Quick source attribution for specific claims |
| **Card carousel** | Below the response body | Curated "recent reads" or "related articles" |
| **Sources button** | Action bar at the bottom | Aggregated source list with stacked favicons |

The card carousel displays 3 cards at a time in a horizontally scrollable row, each linking to an external article with an image, source name, title, and date. **[Confirmed]**

## 2. Architecture Overview

The full pipeline from search query to rendered card:

```
User prompt
  → Model decides to invoke web_search tool
    → Backend search pipeline (Query Fan-Out → Retrieval → RRF → Reranking → Freshness → Safety)
      → Structured results stored in message metadata as `search_result_groups`
        → Conversation object hydrated via GET /backend-api/conversation/{id}
          → Frontend React component reads metadata
            → nav-list-widget renders card carousel
            → webpage-citation-pill renders inline pills
            → Sources button renders aggregated view
```

Key architectural insight from HAR analysis: the conversation object fetched from `/backend-api/conversation/{id}` is the single source of truth. It contains a node graph (`mapping`) where each message node carries both `content.parts[]` (the text) and `metadata` (tool traces, citations, search results, render hints). The frontend is a pure rendering layer over this canonical state. **[Confirmed]** [^1]

## 3. Backend Data Pipeline

When ChatGPT uses its web search tool, the query goes through a multi-stage retrieval pipeline.

> **Provenance note**: The pipeline described below is reconstructed from third-party SEO community analysis [^2] [^3], leaked/public dev console config observations [^11], and public API behavior testing [^4]. None of these stages are officially documented by OpenAI. The public Responses API's web search behavior may differ from ChatGPT's internal search pipeline. Individual stages should be treated as **Likely** unless noted otherwise.

### 3.1 Query Fan-Out
The model doesn't search the user's exact query. It rewrites it into several alternative query variations covering different phrasings, scopes, and intents to widen coverage. **[Likely]** [^2] [^3]

The public API confirms this indirectly: the `web_search_call` output item includes a `search` action with a `queries` field (plural), suggesting multiple queries are generated per search invocation. [^5]

### 3.2 Multi-Provider Retrieval
Each query variation is sent to multiple licensed search providers and APIs. Results are fetched using both keyword matching and semantic embeddings (vector-based similarity). **[Likely]** [^3]

### 3.3 Reciprocal Rank Fusion (RRF)
Only the top results from each provider are collected and merged using RRF, which rewards agreement across sources. Pages appearing highly across multiple retrieval feeds rise to the top. **[Likely]** [^2] [^3]

### 3.4 Neural Reranking
A neural ranking model reshuffles results based on quality signals: structure, depth, clarity, authority, and freshness. The Singularity Digital analysis identified `ret-rr-skysight-v3` as the reranking model name from a ChatGPT dev console config observation [^3] [^11]. **[Likely]** — the model name is from a third-party config observation, not official documentation.

### 3.5 Content Expansion
The model can expand results using two mechanisms: [^4]
- **`open()`**: Fetches larger text slices from the same page at specific line numbers (like scrolling)
- **`click()`**: Follows outgoing links to retrieve content from new pages

> **Important distinction**: The `open()` and `click()` mechanisms were documented by DEJAN through testing the public Assistants/Responses API [^4], not ChatGPT's internal pipeline. The public API equivalents are `open_page` and `find_in_page` actions in the `web_search_call` output [^5]. ChatGPT's internal search pipeline may use similar but not identical mechanisms. **[Confirmed]** for the public API; **[Inferred]** that ChatGPT's internal pipeline uses the same mechanisms.

### 3.6 Answer Assembly
The model compiles selected sources into a response with inline citations. The backend also generates the `search_result_groups` metadata that powers the card carousel. **[Confirmed]** that `search_result_groups` exists [^1]; **[Inferred]** that it is generated at this stage.

### 3.7 Carousel Title Generation

The carousel title (e.g., "Recent reads on BTC liquidity and flows") appears to be contextually generated text included in the structured metadata. In the HTML, it renders as a plain `<div>` within the `nav-list-widget` container, not as parsed markdown from the response body. This suggests the title is either: (a) generated by the model as a separate metadata field alongside `search_result_groups`, or (b) a dedicated field within the `search_result_groups` structure itself. There is no evidence of markdown interception or frontend-side generation. **[Inferred]**

### 3.8 Card vs. Citation Selection Criteria

Not all cited sources appear in the carousel, and not all carousel cards appear as inline citations. The selection criteria for each UI surface likely differ:

- **Inline citation pills**: Tied to specific claims in the text, selected by the model during answer generation
- **Card carousel**: Likely a curated subset based on: relevance scoring, presence of Open Graph metadata (articles with images make better cards), content type (news articles vs. reference pages), and a maximum card count
- **Sources button**: Aggregates all sources consulted

The carousel appears to favor news articles and long-form content with rich metadata over reference pages, tool documentation, or forums. **[Inferred]** — no direct evidence of the selection algorithm has been found.

## 4. Conversation Object & Metadata

### 4.1 Canonical Schema

The conversation object returned by `GET /backend-api/conversation/{id}` follows this structure: **[Confirmed]** [^1]

```
Conversation {
  id
  title
  current_node: NodeId
  mapping: { NodeId: Node }
}

Node {
  id
  parent: NodeId | null
  children: NodeId[]
  message: Message | null
}

Message {
  id
  author: { role: "user" | "assistant" | "system" | "tool" }
  content: { parts: string[] | structured_blocks[] }
  metadata: {
    model_slug?
    citations?
    search_result_groups?        ← Powers the card carousel
    invoked_resource?
    async_task_id?
    ...feature flags and render hints...
  }
}
```

> **Stability caveat**: The HAR study's own validation notes that "'Exact shape of conversation state' as a stable contract" is "Overstated" — the schema is a snapshot in time and may change without notice [^1].

### 4.2 Key Metadata Fields

- **`search_result_groups`**: Server-returned groups of web results used by the search tool. The field name is **Confirmed** from HAR observation [^1]. Its internal structure (containing per-result URL, title, Open Graph image URL, publication date, domain/favicon info) is **Inferred** from what the rendered HTML consumes — the HAR study confirms the field's existence but does not document its sub-fields.
- **`citations`**: Reference objects for inline citation pills, mapping text ranges to source URLs. **[Confirmed]** [^1]

### 4.3 Tool Ecosystem Separation

The HAR analysis reveals a clean separation in the tool ecosystem: **[Confirmed]** [^1]
- **`/ecosystem/widget`** → Returns HTML payload for sandboxed widget rendering (used for shopping, etc.)
- **`/ecosystem/call_mcp`** → Executes named tools against connector URIs

The card carousel does **not** use the widget/iframe sandbox path. It is rendered natively by the React SPA using data from the conversation metadata. **[Confirmed]** [^1]

### 4.4 Shopping vs. Search Widget Distinction

The HAR analysis confirms that shopping results use a fundamentally different rendering path: **[Confirmed]** [^1]

| Feature | Search cards (nav-list-widget) | Shopping widgets |
|---|---|---|
| Rendering | Native React from conversation metadata | Sandboxed iframe via `/ecosystem/widget` |
| Data source | `search_result_groups` in message metadata | Tool output from `/ecosystem/call_mcp` |
| Images | Proxied through `images.openai.com` | Fetched directly from merchant CDNs |
| Sandbox | `oaiusercontent.com` | N/A (native) |

Other widget types (maps, product comparisons, etc.) likely also use the `/ecosystem/widget` iframe path, as ChatKit [^6] provides a general-purpose widget framework (Card, ListView, etc.) for structured content. **[Likely]**

### 4.5 Streaming and SSE Behavior

ChatGPT uses SSE streaming for response text delivery and WebSocket (bootstrapped via `celsius` → `wss://ws.chatgpt.com`) for realtime updates. **[Confirmed]** [^1]

The card carousel renders from `search_result_groups` metadata attached to the message node. This metadata is most likely populated either:
1. As part of SSE tool-call result events during streaming, arriving when the search tool completes
2. As a post-stream metadata update event after the response finishes

The `data-scroll-anchor="true"` attribute on the conversation turn container is likely for auto-scrolling during text streaming, not carousel-specific. The carousel most likely appears after search results are fully assembled, since it requires structured metadata (images, dates, titles) that would be resolved server-side. **[Inferred]**

## 5. Image Proxying Strategy

Article images are never fetched directly from publisher domains in the chat UI. Two proxying strategies are used:

### 5.1 Article Images

Proxied through OpenAI's CDN: **[Confirmed]** from live HTML
```
https://images.openai.com/static-rsc-1/{opaque-hash-token}
```

> **Path analysis**: The `static-rsc-1` prefix is an opaque CDN path observed in live HTML. The meaning of "rsc" is **Unverified** — plausible interpretations include "React Server Components" (given ChatGPT's Next.js stack), "Resource," or an internal naming convention. The `-1` suffix may indicate a versioned or rotating endpoint (e.g., `-2`, `-3` in the future). This path is not publicly documented and should be treated as potentially unstable. **[Confirmed]** that it exists; **[Unverified]** regarding its meaning, stability, and rotation behavior.

Benefits:
- **Security**: No third-party tracking pixels or cookies loaded in chat context
- **Availability**: Images remain accessible even if source site changes URLs
- **Performance**: Served from OpenAI's CDN with caching
- **Privacy**: User's IP is not exposed to article publishers

A secondary proxy path exists for thumbnails: **[Confirmed]** from HAR [^1]
```
https://images.openai.com/thumbnails/{path}
```

### 5.2 Favicons

Uses Google's public favicon service at multiple size tiers: **[Confirmed]** from live HTML
```
https://www.google.com/s2/favicons?domain={domain}&sz={size}
```

| Size | Usage |
|---|---|
| 32px | Inline source label next to domain name |
| 128px | Centered favicon on cards without article images |
| 256px | Blurred background layer on cards without article images |

## 6. Frontend Component Anatomy

> **CSS class provenance note**: ChatGPT's frontend uses Tailwind CSS with a custom design system. Classes prefixed with `token-` (e.g., `text-token-text-primary`, `bg-token-bg-primary`, `border-token-border-default`) are **ChatGPT's custom design tokens** backed by CSS variables (`--token-*`). Several other utility classes are also custom — see Section 6.7 for the full breakdown.

### 6.1 Container Structure

```html
<div class="mb-1 mt-6" data-testid="nav-list-widget">
  <!-- Dynamic title generated by the model -->
  <div class="text-token-text-primary -mb-1 pb-3 font-semibold">
    Recent reads on BTC liquidity and flows
  </div>

  <div class="relative">
    <!-- Horizontal scroll container with CSS snap -->
    <div class="flex flex-row gap-4 py-2 no-scrollbar
                snap-x snap-mandatory overflow-x-scroll scroll-smooth">
      <!-- Cards -->
    </div>

    <!-- Navigation arrows overlay -->
    <div class="pointer-events-none absolute start-0 top-0
                flex h-full w-full flex-row">
      <!-- Left arrow (opacity: 0 when at start) -->
      <!-- Right arrow (opacity: 1 when more cards exist) -->
    </div>
  </div>
</div>
```

Key CSS details:
- `snap-x snap-mandatory` for snapped horizontal scrolling **(standard Tailwind)**
- `no-scrollbar` hides the native scrollbar **(custom utility — common community pattern)**
- `scroll-smooth` for animated scroll on arrow click **(standard Tailwind)**
- Navigation arrows use `pointer-events-none` on the container, with `pointer-events-auto` only on the buttons **(standard Tailwind)**

### 6.2 Card Layout

Each card occupies exactly 1/3 of the visible width:

```css
basis-[calc((100%-2rem)/3)]
```

This accounts for the `gap-4` (1rem) between cards: 3 cards + 2 gaps = `100% - 2rem` divided by 3.

Cards use `shrink-0 snap-start` to prevent flex shrinking and snap to the left edge.

### 6.3 Card Variants

**Variant A: With Article Image (Open Graph image available)**

```html
<div class="relative h-36 w-full overflow-hidden">
  <img class="m-0 h-full w-full"
       alt="Article title"
       src="https://images.openai.com/static-rsc-1/{hash}"
       style="object-fit: cover;">
</div>
```

**Variant B: Favicon-Only Fallback (no Open Graph image)**

```html
<div class="icon-bg relative flex h-36 w-full shrink-0
            items-center justify-center overflow-hidden">
  <!-- Blurred 256px favicon as ambient background -->
  <div class="absolute inset-0 flex items-center justify-center
              bg-cover bg-center bg-no-repeat blur-3xl">
    <img src="https://www.google.com/s2/favicons?domain={domain}&sz=256"
         class="mix-soft-light opacity-30 dark:opacity-20">
  </div>
  <!-- Centered 128px favicon -->
  <div class="absolute flex h-10 w-10 items-center justify-center">
    <img src="https://www.google.com/s2/favicons?domain={domain}&sz=128"
         class="icon-xl rounded-full">
  </div>
</div>
```

The `mix-soft-light` blend mode + `blur-3xl` on the large favicon creates a brand-colored ambient backdrop, which is a clever fallback when no article image was scraped.

> **Note**: `mix-soft-light` is a **custom ChatGPT utility**, not standard Tailwind. Standard Tailwind provides `mix-blend-soft-light` for the same `mix-blend-mode: soft-light` effect. ChatGPT likely has an alias or shorthand plugin. Similarly, `icon-bg` and `icon-xl` are custom design system classes (see Section 6.7).

### 6.4 Card Body (Common to Both Variants)

```html
<div class="flex flex-col gap-2 overflow-hidden px-4 text-ellipsis">
  <!-- Source: favicon + domain -->
  <div class="text-token-text-primary flex items-center gap-1.5 text-xs">
    <img src="https://www.google.com/s2/favicons?domain={domain}&sz=32"
         class="icon-xs my-0 rounded-full">
    Reuters
  </div>
  <!-- Title: clamped to 5 lines -->
  <div class="text-token-text-primary decoration-token-link
              line-clamp-5 text-sm font-medium">
    Global crypto ETFs attract record $5.95 billion as bitcoin scales new highs
  </div>
  <!-- Publication date -->
  <div class="text-token-text-secondary text-xs">Oct 7, 2025</div>
</div>
```

### 6.5 Card Wrapper

Each card is an `<a>` tag wrapping the image + body:

```html
<a href="https://www.reuters.com/...?utm_source=chatgpt.com"
   target="_blank"
   rel="noopener"
   class="flex flex-col items-center gap-4 overflow-hidden pb-6">
```

All outbound URLs have `?utm_source=chatgpt.com` appended for attribution.

### 6.6 Card Outer Shell

```html
<div class="h-full w-full bg-token-bg-primary cursor-pointer
            transition-all duration-200
            border-token-border-default hover:border-light
            border rounded-xl bg-clip-padding
            hover:shadow-md overflow-hidden">
```

The `hover:shadow-md` + `hover:border-light` provide a subtle lift-on-hover effect. `bg-clip-padding` ensures the background doesn't bleed through the border.

> **Note**: `border-light` is a **custom ChatGPT color token**, not a standard Tailwind class. Standard Tailwind uses `border-{color-name}` with its default palette.

### 6.8 Responsive Behavior

The card layout uses `basis-[calc((100%-2rem)/3)]` for a 3-column desktop layout. The parent thread content area uses Tailwind v4 container query variants (`@w-sm/main:` and `@w-lg/main:` syntax, where `main` is the named container) for responsive margins. **[Confirmed]** from live HTML.

The specific mobile behavior of the carousel was not directly observed, but it is **Likely** that:
- On smaller containers, the `basis-[calc(...)]` value changes to show fewer cards (1-2)
- The horizontal scroll with `snap-x` naturally adapts to any card count
- The CSS snap behavior provides a mobile-friendly swipe experience
- Container queries (not viewport media queries) control the adaptation, consistent with ChatGPT's component-scoped responsive approach

### 6.9 Accessibility

Based on the observed HTML, the carousel has several accessibility characteristics and gaps:

**Present:**
- Cards are `<a>` tags (inherently keyboard-focusable and screen reader-accessible)
- Links have `target="_blank"` and `rel="noopener"` for safe external navigation
- Images have `alt` text (article titles)

**Absent (accessibility gaps):**
- No `role="region"` or `aria-roledescription="carousel"` on the container
- Navigation arrow buttons have no visible `aria-label` in the HTML
- `no-scrollbar` hides the scrollbar, reducing visual affordance for scroll discovery
- No visible focus indicator styling customization observed
- No live-region announcements for carousel navigation

**Confidence**: **[Confirmed]** from HTML inspection. These gaps may be addressed by JavaScript-injected ARIA attributes not visible in the static HTML snapshot.

## 7. Inline Citation Pills

Separately from the card carousel, inline citation pills (`data-testid="webpage-citation-pill"`) are embedded within paragraph text:

```html
<span class="ms-1 inline-flex max-w-full items-center select-none
             relative top-[-0.094rem]
             animate-[show_150ms_ease-in]"
      data-testid="webpage-citation-pill"
      style="width: 75px;">
  <a href="https://..."
     class="flex h-4.5 overflow-hidden rounded-xl px-2
            text-[9px] font-medium
            text-token-text-secondary!
            bg-[#F4F4F4]! dark:bg-[#303030]!">
    <!-- Primary source (visible) -->
    <span class="flex h-4 w-full items-center justify-between"
          style="opacity: 1; transform: none;">
      <span class="max-w-[15ch] truncate">Lyn Alden</span>
      <span class="-me-1 ... text-[#8F8F8F]">+2</span>
    </span>
    <!-- Secondary source (hidden, animates in) -->
    <span class="flex ... absolute"
          style="opacity: 0; transform: translateX(10%);">
      <span class="max-w-[15ch] truncate">21shares</span>
      <span>+2</span>
    </span>
  </a>
</span>
```

Key details:
- 9px font, `h-4.5` height (custom), `rounded-xl` for pill shape
- `max-w-[15ch]` truncates long source names
- **Source rotation is React state-driven**: The inline `style` attributes (`opacity: 0; transform: translateX(10%)`) are characteristic of React-controlled transitions, not pure CSS keyframes. React state changes toggle which source span is visible by updating inline styles. The initial pill appearance uses a CSS keyframe animation (`show` keyframe via `animate-[show_150ms_ease-in]`), but the rotation between sources is JavaScript-driven. **[Confirmed]** from inline style patterns — pure CSS animations would use `animation` or `transition` properties in the class, not inline `style` attributes.
- "+N" badge shows additional source count
- `top-[-0.094rem]` for precise baseline alignment with surrounding text

## 9. Public API vs Internal API

### 9.1 Public API (Responses API with `web_search` tool)

The public API returns minimal citation data: **[Confirmed]** [^5]

```json
{
  "type": "url_citation",
  "start_index": 2606,
  "end_index": 2758,
  "url": "https://...",
  "title": "Title..."
}
```

Available fields: `url`, `title`, `start_index`, `end_index`. No images, no dates, no favicon data, no grouping. **[Confirmed]** [^5]

Additionally, the API provides:
- **`sources` field**: Available via `include=["web_search_call.action.sources"]`. Returns the complete list of URLs the model consulted — often more URLs than cited. Returns URL-only data, not enriched metadata. Real-time third-party feeds (sports, weather, finance) are also surfaced here with `oai-` labels. **[Confirmed]** [^5]
- **`web_search_call` actions**: The output includes action types (`search`, `find_in_page`, `open_page`) with the `search` action sometimes including the `queries` that were searched. **[Confirmed]** [^5]

### 9.2 Internal API (ChatGPT web client)

The internal conversation object includes richer metadata: **[Confirmed]** from HAR [^1]
- `search_result_groups` with full article metadata (images, dates, domains) **[Inferred]** — field exists, sub-structure inferred from rendered output
- Grouping and relevance scoring for UI treatment decisions **[Inferred]**
- Proxied image URLs through `images.openai.com` **[Confirmed]** from live HTML

### 9.3 Implications

The card carousel is a **ChatGPT-specific frontend feature** that cannot be replicated using the public API alone. To build something similar, you would need to:
1. Extract URLs from `url_citation` annotations (and optionally from `sources` for the full list)
2. Fetch Open Graph metadata yourself (title, image, date, site name)
3. Proxy images through your own CDN
4. Render your own carousel component

### 9.4 Vercel AI SDK Comparison

The Vercel AI SDK Elements library (v1.8.4, February 2026) [^10] provides two related components:

| Component | Equivalent to | Description |
|---|---|---|
| `InlineCitation` | Inline citation pills | Pill trigger + hover card with source details and carousel for multiple sources |
| `Sources` | Sources button | Trigger button + collapsible panel listing all sources |
| _(none)_ | Card carousel (nav-list-widget) | **No equivalent exists** |

Key differences:
- The `InlineCitation` component uses `experimental_useObject` for structured citation data, not the streaming `useChat` integration. The AI SDK docs explicitly state: "there is no official support for inline citations with Streamdown or the Response component" [^10].
- The `Sources` component works with `useChat` and the `sendSources: true` option in `toUIMessageStreamResponse()`.
- Neither component fetches or displays article images, publication dates, or Open Graph metadata — they work with URL + title data only.
- There is **no card carousel component** in AI SDK Elements. Implementing a ChatGPT-like carousel in Not A Wrapper would require a custom component. **[Confirmed]** from AI SDK documentation.

## 10. Design Patterns Worth Adopting

### 10.1 Favicon-as-Ambient-Background
Using a blurred, oversized favicon with `mix-blend-mode: soft-light` as a fallback when no article image is available. Creates a brand-colored backdrop without requiring any additional asset.

### 10.2 CSS Snap Scroll Carousel
`snap-x snap-mandatory` with `snap-start` items and `scroll-smooth` provides a native, performant carousel without JavaScript scroll handling.

### 10.3 Image Proxying
Routing all external images through a proxy CDN for security, privacy, caching, and availability.

### 10.4 Citation Pill Source Rotation
React state-driven transitions between multiple source names in a fixed-width pill, showing the user that multiple sources support a claim without consuming more horizontal space. Uses inline style toggling for smooth opacity + translateX transitions.

### 10.5 Stacked Favicon Circles
Using `flex-row-reverse` + negative margins + matching border color for a clean overlapping avatar pattern.

### 10.6 UTM Attribution
Appending `?utm_source=chatgpt.com` to all outbound links for publisher analytics.

### 10.7 Metadata-Driven UI Surfaces
Using a single structured metadata payload (`search_result_groups`) to power three distinct UI surfaces (carousel, pills, sources button) with different selection criteria and visual treatments. This pattern decouples data from presentation effectively.

## 11. Open Questions

The following questions remain unanswered after research:

1. **`search_result_groups` internal schema**: What are the exact field names and types within this metadata object? HAR observation confirms it exists but does not expose its internal structure. A targeted HAR capture that intercepts the full conversation object payload would be needed to answer this.

2. **Carousel card maximum count**: Is there a hard cap on the number of cards in the carousel? The observed instance showed approximately 6-8 cards, but the limit is unknown.

3. **Title generation mechanism**: Is the carousel title ("Recent reads on BTC liquidity and flows") a separate metadata field, part of `search_result_groups`, or generated by the model as part of the response content that the frontend extracts?

4. **Streaming chronology**: Does the carousel appear during SSE streaming (progressively, as search results arrive) or only after the response is complete? If the former, what SSE event type delivers the `search_result_groups` metadata?

5. **`static-rsc-1` path stability**: Does the CDN path version rotate (e.g., `static-rsc-2`, `static-rsc-3`)? What does "rsc" stand for? Is image availability time-limited?

6. **Mobile breakpoints**: What specific container query breakpoints change the carousel from 3-column to 1-2 column? Are the breakpoints defined in ChatGPT's Tailwind config or computed dynamically?

7. **Map/location widgets**: Do location-based search results produce a different widget type (e.g., embedded map cards), and if so, do they use the `nav-list-widget` pattern or the `/ecosystem/widget` iframe path?

8. **ARIA attribute injection**: Are accessibility attributes (carousel role, arrow labels) injected by React at runtime? The static HTML snapshot shows gaps, but JavaScript may add them dynamically.

9. **Citation pill rotation interval**: How frequently do the citation pills rotate between sources? Is it a fixed interval or user-interaction-driven (e.g., rotating on hover)?

10. **`sources` field enrichment roadmap**: Will the public API's `sources` field eventually return enriched metadata (images, dates, descriptions) that could power a carousel without client-side OG scraping?
